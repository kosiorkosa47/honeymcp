# SLSA Level 3 build provenance

honeymcp release artifacts (binaries + container images) ship with
[SLSA Level 3](https://slsa.dev/spec/v1.0/levels#build-l3) build
provenance. Provenance is generated by the GitHub-native
`actions/attest-build-provenance` action during the release workflow,
signed with sigstore via OIDC, and registered against this repo so
downstream consumers can verify it without trusting any single human
identity.

## What this means

> SLSA L3 means the build platform attests, in a signed predicate, to
> the source repo and commit the artifact came from, and that the build
> ran on a hosted, isolated platform that prevents the operator from
> tampering with the inputs or output.

In practice this gives you four guarantees you can verify in one
command:

1. The artifact was built from a commit on `kosiorkosa47/honeymcp`.
2. The build ran on GitHub-hosted runners (not a developer's laptop).
3. The provenance predicate is signed by a sigstore identity bound to
   the workflow file at the commit it was built from — meaning a
   tampered workflow can't backdate provenance to look legitimate.
4. The signature is publicly verifiable via the sigstore transparency
   log; you don't need to trust any one private key.

## Verifying a binary release

```bash
# 1. Download the artifact and its sha256 from the GitHub release page.
gh release download vX.Y.Z --pattern '*linux-gnu*'

# 2. Verify the build provenance attestation.
gh attestation verify honeymcp-vX.Y.Z-x86_64-unknown-linux-gnu.tar.gz \
    --owner kosiorkosa47

# Output:
# Loaded digest sha256:... for file://honeymcp-...tar.gz
# Loaded 1 attestation from GitHub API
# ✓ Verification succeeded!
```

If `gh attestation verify` fails, **do not deploy the artifact**. The
binary either has a digest mismatch, was uploaded outside the official
release pipeline, or the workflow that produced it doesn't match the
expected workflow ref.

## Verifying a container image

```bash
# Pull the image you intend to deploy.
docker pull ghcr.io/kosiorkosa47/honeymcp:vX.Y.Z

# Verify both the cosign keyless signature AND the SLSA provenance.
cosign verify ghcr.io/kosiorkosa47/honeymcp:vX.Y.Z \
    --certificate-identity-regexp '^https://github\.com/kosiorkosa47/honeymcp/' \
    --certificate-oidc-issuer https://token.actions.githubusercontent.com

gh attestation verify oci://ghcr.io/kosiorkosa47/honeymcp:vX.Y.Z \
    --owner kosiorkosa47
```

Both must succeed. The cosign check covers the image signature; the
attestation check covers the SLSA predicate (build inputs, source
ref, builder identity).

## Verifying the SBOM (independent of provenance)

The release workflow also publishes SPDX + CycloneDX SBOMs as image
attestations:

```bash
cosign verify-attestation \
    --type spdxjson \
    --certificate-identity-regexp '^https://github\.com/kosiorkosa47/honeymcp/' \
    --certificate-oidc-issuer https://token.actions.githubusercontent.com \
    ghcr.io/kosiorkosa47/honeymcp:vX.Y.Z
```

## How this maps to SLSA L3 requirements

| SLSA L3 requirement | How honeymcp satisfies it |
| --- | --- |
| Source — version controlled | Public GitHub repo; signed commits required on `main` |
| Source — verified history | Branch protection enforces signed commits + required reviews |
| Build — scripted | All builds via `.github/workflows/release.yml` |
| Build — build service | GitHub-hosted runners (ubuntu-latest, macos-latest, windows-latest) |
| Build — ephemeral environment | GitHub-hosted ephemeral VMs; no persistent build state |
| Build — isolated | One job, one VM, no cross-job state mutation |
| Provenance — exists | `actions/attest-build-provenance@v3` emits `intoto.jsonl` per artifact |
| Provenance — authenticated | Sigstore OIDC signature bound to workflow file at commit |
| Provenance — service generated | GitHub Actions runner produces the predicate, not user code |
| Provenance — non-falsifiable | Signed by sigstore Fulcio, transparency-logged in Rekor |
| Provenance — dependencies complete | Cargo lockfile + SPDX SBOM record full transitive deps |
| Common — security | Branch protection + required reviews + signed commits + Dependabot |
| Common — access | Push to `main` requires PR + green CI + approval |
| Common — superusers | No bypass roles; admin push to `main` is alarmed in audit log |

## Why we don't claim SLSA L4

L4 requires a hermetic build (no network during compile + reproducible
binaries). honeymcp's Rust build hits crates.io for the Cargo registry
during `cargo fetch`, which fails the hermetic-build test. We could
satisfy this with `cargo vendor` + a registry mirror, but the cost
isn't worth the marginal trust improvement for a project at this
scale. We'll revisit if a buyer asks.

## Replaying provenance on a CVE

If a downstream user reports a CVE in a dependency:

1. Pull the SLSA provenance for the affected release: `gh attestation list --owner kosiorkosa47 --repo honeymcp`.
2. Read the `materials` array in the predicate to identify the exact
   commit the artifact was built from.
3. Cross-reference against the SBOM (also attested) to confirm the
   vulnerable dependency version was actually present.
4. Use `cargo deny check advisories` against that commit to validate
   the report.

This sequence takes ~2 minutes per CVE and produces a defensible
record that survives audit.
